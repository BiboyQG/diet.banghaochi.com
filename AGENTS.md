# AGENTS.md

## Workflow

- Ask for clarification before acting when the request is ambiguous or could affect production data, Cloudflare configuration, GitHub publishing, or iOS signing.
- Keep changes scoped. Remove dead code, unused imports, generated scratch files, and temporary scaffolding before finishing.
- Do not commit secrets. Keep Cloudflare API tokens, Access service tokens, `.dev.vars`, `.env*`, signing credentials, D1 exports, and auth cookies out of git.
- Use `jq` for JSON and `yq` for YAML/XML when shell parsing is needed.
- Use `python3`; there is no guaranteed `python` binary. Prefer repo/node tooling when it already exists.
- Use `apply_patch` for manual text edits. Generated image/icon resizing and formatter output may be produced by their normal tools.

## Project Shape

- Web: `apps/web`, Vite + React + TypeScript.
- Worker/API: `apps/worker`, Hono + Cloudflare Workers + D1.
- Shared schemas/calculations: `packages/shared`.
- iOS: `apps/ios`, SwiftUI app generated by XcodeGen.
- Production host: `https://diet.banghaochi.com`.
- Staging host: `https://diet-staging.banghaochi.com`.
- Reusable food templates live in D1 `food_templates` and must stay aligned across Worker routes, web API/types, iOS models/client/store, and shared schemas.

## Auth

- Current v1 auth is Cloudflare Access One-time PIN allowlisted to `m13971212844@gmail.com`.
- iOS uses `ASWebAuthenticationSession` via `/auth/ios-callback`; the Worker passes the Access token back through the `diettracker://access/callback` URL.
- Do not embed service tokens in the iOS app or web bundle.
- Passkeys are not a drop-in replacement for Cloudflare Access OTP here. A passkey path would mean either configuring an external passkey-capable IdP for Access or replacing Access with app-owned WebAuthn/passkey auth plus session middleware.

```bash
npm run db:migrate:local
npm run dev
```

The web dev server runs at `http://127.0.0.1:5173` and proxies `/api` to the Worker at `http://127.0.0.1:8787`.

## Checks

Run the narrowest relevant checks first, then broaden before publishing or deploying:

```bash
npm run typecheck
npm run lint
npm test
npm run test:worker
npm run test:web
npm run test:e2e
```

For interface animation changes, keep motion subtle, preserve reduced-motion behavior, and add focused tests for any stateful web feedback. For SwiftUI-only animation modifiers, simulator tests are the main guard unless data flow changes.

For iOS, prefer XcodeBuildMCP when available. The CLI equivalent is:

```bash
cd apps/ios
xcodegen generate --spec project.yml
xcodebuild test -project NutritionTracker.xcodeproj -scheme NutritionTracker -destination 'platform=iOS Simulator,name=iPhone 17'
```

## Deployment

- Apply D1 migrations before deploying an environment.
- Production/staging D1 database IDs are in `apps/worker/wrangler.jsonc`; do not replace them with placeholders during normal work.
- Deploy with:

```bash
cd apps/worker
npx wrangler d1 migrations apply DB --env staging --remote
npx wrangler deploy --env staging
npx wrangler d1 migrations apply DB --env production --remote
npx wrangler deploy --env production
```
